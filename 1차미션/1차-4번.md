# 1.
[학습 과정]
사용되는 기술의 이해를 위해 공식 문서와 코드를 직접 파고들며 원리를 파악하는 방식으로 학습하고, 직접 코드 작성으로 재현한 뒤 이해한 것을 문서로 기록합니다. 이렇게 정리한 내용은 블로그와 팀원들에게 꾸준히 공유해 기술 공통 관심사를 키우고 있으며, 이 과정을 약 3년간 지속하여 문제의 해결 자세로 삼고 있습니다. 최근에는 GPT 기반 퀴즈를 활용해 개념을 스스로 설명하고 검증하는 단계를 더해 정확도를 높이고 있습니다.

대표적으로, 서비스마다 동일한 API를 호출했음에도 TCP 연결 유지 시간이 서로 다르게 나타난 현상이 있었고, 단순히 타임아웃 시간을 늘리는 대신, RestTemplate과 내부 구현체의 연결 관리 방식을 직접 분석한 경험이 있습니다. keep-alive 정책과 설정 차이가 원인이었음을 공유했고, 이를 바탕으로 서비스 전체 연결 정책을 통합할 수 있었습니다.

이 경험은 특정 문제 하나를 해결하는 데 그치지 않고, 분석-재현-기록-전파라는 동일한 방식을 다양한 기술 주제에 적용하는 기반이 됐습니다. 최근 겪었던 MyBatis의 ResultMap으로 RDBMS 강결합을 줄이기나, 새로운 비회원 인증 체계 설계 등 여러 상황에 대해 동일 패턴으로 정리했으며, 새로운 이슈에서도 빠르게 원인을 좁혀 개선안 제안을 할 수 있게 되었습니다.

[역량]
- 새로운 기술을 빠르게 습득하고 문서로 정리하는 학습 능력
- 동작 구조 분석 및 재현을 통해 원인을 찾는 문제 해결 역량
- 동료들과 지식을 나누며 함께 성장할 수 있는 협업 및 커뮤니케이션 능력

[목표]
앞으로도 학습, 검증, 문서화를 중심으로 한 방식을 바탕으로 기술적 의사 결정에 주도적인 참여와 서비스 안정성과 API 운영 기준 등 핵심 구조를 개선하고 싶습니다.
단순 이슈를 해결하는 데 그치지 않고, 과정에서 얻은 인사이트 기준과 가이드로 정리해 재사용할 수 있도록 만드는 개발자가 되어 조직의 신뢰도와 서비스 품질을 꾸준히 높이는 데 기여하고자 합니다.

# 2.
백엔드의 매력은 같은 요청과 응답을 처리하더라도 그 과정에서 트랜잭션 경계, 데이터 정합성, 자원 사용 전략에 따라 서비스 성능과 안정성이 완전히 달라진다는 점이라고 생각합니다. 사용자는 동일하게 이용하더라도 어떤 방식으로 쿼리를 설계하고 외부 연동과 장애를 격리하느냐에 따라 전체 경험이 달라지기 때문에, 더 잘 만들기 위해 끊임없이 배우고 구조를 개선하는 역할에 큰 흥미와 동기를 느낍니다.
특히 장애를 끝까지 추적하여 로그와 지표, 커뮤니티 반응을 보며 해소했을 때 곧바로 에러율, 문의 건수 같은 숫자의 감소 순간이 재밌으며 이런 성향이 백엔드 역할과 잘 맞다고 판단하였습니다.

대표 사례로, 무버스 서비스에서 고객이 차량 반납 시 촬영한 이미지를 전송하는 과정에서 문제가 반복된 적이 있습니다. 10장 이상의 이미지를 S3에 업로드할 때 리드 타임아웃이 자주 발생해, 고객이 여러 번 재시도하며 서버 부하가 커지고 고객은 반납 이후 프로세스 처리가 불가능했습니다.
원인을 분석해 보니, 서버가 이미지를 직접 S3에 동기적으로 업로드하는 구조가 병목이었고, 개선하기 위해 AWS S3 Presigned URL 방식을 학습해 이미지와 반납 처리를 분리하도록 프로세스를 제안하고 적용했습니다.
클라이언트가 이미지를 직접 업로드 후 반납 처리하도록 역할을 나눠 서버 리소스를 줄였으며, 관련 문의가 0건으로 감소한 것을 경험했습니다.

이 경험으로 백엔드 개발은 기능 구현을 넘어 트래픽 흐름과 자원 사용의 고민이 불편함 없이 서비스를 이용하도록 만드는 역할이라는 것을 다시 느꼈습니다. 앞으로도 백엔드를 커리어의 중심으로 장애가 커뮤니티 반응이나 문의로 나오기 전에 모니터링, 알림 체계와 성능 및 리소스 튜닝을 통해 사전에 감지, 완화하는 데 집중하고자 합니다. 또한 장애 발생의 전파 범위를 최소화하여 개발자와 운영 조직이 반복적인 장애 대응에 쓰는 시간을 줄이고, 고객은 불편 없이 서비스를 이용할 수 있도록 전체 플로우를 설계 및 개선하는 개발자로 성장하고 싶습니다.

# 3.
경험을 단순 소비하고 지나가는 것이 아닌, 이해한 내용을 스스로의 언어로 정리해 '나만의 개인 기술 도서관'을 만들고 싶다는 생각으로 블로그를 시작했습니다. 처음에는 개인 기록으로 편하게 작성했으나, 3년 가까이 약 200개 글을 작성하며 학습 방식과 글을 대하는 태도가 바뀌었습니다.

초기에는 단순 에러 해결 방법을 적는 수준에서 방문자가 꾸준히 늘고 도움 되었다는 피드백을 받으면서 자연스럽게 책임감을 느끼게 되었습니다. 이를 계기로 단순 해결책 나열보다 문제 발생 이유, 여러 선택지 중 특정 방식을 선택한 이유, 직접 적용해 보며 얻은 인사이트 등을 함께 정리하려고 노력했습니다. 이 과정에서 글의 기승전결 고민과 목차, 소제목, 코드 블록 등을 활용하여 처음 보는 사람도 쉽게 읽을 수 있는 글의 구조에도 신경 쓰게 되었습니다.

대표적으로 Apache POI 기반 엑셀 생성에서 OutOfMemory 문제가 반복되었을 때, 단순 문제 해결에 그치지 않고 '어떻게 설명할 것인가'까지 함께 고민하여 접근했습니다. 문제 발생 배경부터 XSSF, SXSSF의 동작 방식, 최종 선택 구조와 그에 따른 효과를 한눈에 보이도록 글을 작성해 팀원들에게 발표했고, 덕분에 빠른 이해와 참고할 수 있는 문서로 자리 잡았습니다. ""원리까지 정리되어 훨씬 이해가 쉽다""는 팀원의 피드백을 듣고, 이해할 수 있는 형태로 전달하는 것의 중요성을 알게 되었습니다.

돌아보면, 블로그는 단순한 기록 공간이 아닌 몰입을 만들어주는 장치였습니다. 하나의 글을 완성하기 위해서 공식 문서와 코드를 다시 확인하고, 이해가 어려운 부분은 예제 프로젝트를 만들어 검증하는 과정이 반복되며 기술 자체에 대한 이해도와 글쓰기 역량이 함께 성장했습니다.
초창기와 최근 글을 비교했을 때, 문제 정의 방식, 설명의 깊이, 글 구조가 눈에 띄게 달라졌음을 느꼈습니다.

지금은 새로운 경험을 할 때, 하고 끝내는 것이 아닌 무엇을 배웠고, 다음에는 어떻게 더 잘할 수 있을지까지 정리하는 습관으로 자리 잡았습니다.
